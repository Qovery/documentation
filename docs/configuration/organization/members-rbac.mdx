## Organization Members

You can invite someone to join your organization by email. Once they accept the invitation, they will have access to the organization and all its projects based on their assigned role.

### Accessing Members Settings

<Steps>
  <Step title="Open Organization Settings">
    Click the Settings tab while you are on your organization overview.

    <Frame>
      <img src="/images/configuration/organization/access_settings.png" alt="How to access your organization settings" />
    </Frame>

  </Step>

  <Step title="Navigate to Members Section">
    Open the **Members** section within the organization settings

    <Frame>
      <img src="/images/configuration/organization/members.png" alt="Qovery - List all members within an organization" />
    </Frame>

  </Step>
</Steps>

### Inviting Members

1. Click the **Invite Member** button
2. Enter the member's email address
3. Select a role (see Default Roles below)
4. Click **Send Invitation**

The invited member will receive an email with instructions to join the organization.

<Info>
  Changing the role of a member requires the user to logout/login to make the
  changes effective or wait a few minutes (max 1 hour).
</Info>

## Default Roles

Qovery provides 5 built-in roles. Each user can only hold **one** role at a time.

### Owner

Full access to the organization. Only one user can be the owner, and ownership can be transferred (see below).

### Admin

Same as Owner except cannot delete the organization or transfer ownership.

### DevOps

Can manage organization infrastructure (clusters, registries, webhooks, API tokens) and has full access to all projects and environments across the organization.

### Billing Manager

Can only manage billing. Has read-only access to clusters but no access to any project.

### Viewer

Read-only access to all organization resources, clusters, and projects.

## Default Roles — Permissions Matrix

| Action | Owner | Admin | DevOps | Billing Manager | Viewer |
| --- | :---: | :---: | :---: | :---: | :---: |
| Read organization data | yes | yes | yes | yes | yes |
| Edit organization settings | yes | yes | no | no | no |
| Delete organization | yes | no | no | no | no |
| Transfer ownership | yes | no | no | no | no |
| Manage billing | yes | yes | no | yes | no |
| Manage members & roles | yes | yes | no | no | no |
| Manage clusters & container registries | yes | yes | yes | no | no |
| Manage organization setup (API tokens, webhooks, SSO) | yes | yes | yes | no | no |
| Create project | yes | yes | yes | no | no |
| Full access to ALL projects & environments | yes | yes | yes | no | no |
| Read-only access to ALL projects & environments | yes | yes | yes | no | yes |
| Read cluster info | yes | yes | yes | yes | yes |

<Info>
  **DevOps vs Admin:** The main difference is that Admin can manage members/roles and billing, while DevOps cannot. Both have full access to all projects and environments, and both can generate API tokens.
</Info>

## Custom Roles

Custom roles let you grant access scoped to specific projects and clusters, instead of giving access across the entire organization.

### What custom roles can and cannot do

Custom roles are configured through **cluster-level** and **project-level** permissions only. They **always** inherit the ability to read organization data (registries, clusters, label groups), which is necessary to work with Qovery resources.

**Custom roles cannot:**
- Generate API tokens
- Manage organization settings (webhooks, SSO)
- Manage members & roles
- Manage billing
- Access projects or clusters beyond what is explicitly granted

<Warning>
  Because custom roles cannot generate API tokens, a user with a custom role cannot run `qovery token`. To use the Terraform provider, an org admin must generate an API token on their behalf, or the user must temporarily be assigned the built-in **DevOps** role.
</Warning>

### Cluster Level Permissions

Set per cluster (or for all clusters at once):

| Level | What it allows |
| --- | --- |
| **Read-Only** | View cluster information (name, region, status). Minimum level. |
| **Create Environment** | Create and manage environments on this cluster. Cluster settings (nodes, type) remain read-only. Required for `terraform apply` when managing environments. |
| **Full Access** | Everything in Create Environment, plus manage cluster settings (start/stop, node scaling, configuration). |

### Project Level Permissions

Set per project, independently for each environment type (Development, Preview, Staging, Production):

| Level | What it allows |
| --- | --- |
| **No Access** | The user cannot see or interact with this environment type. |
| **Read-Only** | View environments, services, logs, and variables. No write access. |
| **Deploy** | Everything in Read-Only, plus trigger deployments, stop services, and connect via shell. Cannot modify service configuration or environment variables. |
| **Manage** | Everything in Deploy, plus edit service configuration, manage environment variables and secrets, create and delete services. Required for `terraform apply`. |
| **Full Access** | Everything in Manage, plus delete environments and manage project-level settings. Equivalent to project admin. |

### Comparing a custom role to the built-in DevOps role

A common question is whether a custom role with **Full Access** on clusters and **Full Access** on all projects is equivalent to the DevOps role.

| Capability | DevOps (built-in) | Custom role (Full Access clusters + Full Access all projects) |
| --- | :---: | :---: |
| Full access to all clusters | yes | yes (if set on all clusters) |
| Full access to all projects & environments | yes | yes (if set on all projects) |
| Read org data (registries, label groups) | yes | yes |
| Generate API tokens | yes | **no** |
| Manage org setup (webhooks, SSO) | yes | **no** |
| Manage members & roles | no | no |
| Manage billing | no | no |
| Scoped to specific projects only | no | **yes** |

**Key takeaway:** A custom role with Full Access everywhere is close to DevOps for day-to-day operations, but cannot generate API tokens or manage org-level settings. If a user needs to generate a Terraform API token themselves, they need the built-in **DevOps** role (or Admin/Owner).

### Minimum custom role for Terraform

To run `terraform plan` and `terraform apply` against a specific project:

| Scope | Minimum level |
| --- | --- |
| Cluster (hosting the target environment) | **Create Environment** |
| Project — target environment type | **Manage** |
| Project — other environment types | No Access (or higher if needed) |
| Other projects | No Access |

The API token itself must be generated by an org admin or a DevOps user and shared with the custom-role user as `QOVERY_API_TOKEN`.

## Transfer Ownership

You can transfer ownership of the organization to another member. To do so:

1. Navigate to the **Members** section in organization settings
2. Click on the member you want to transfer ownership to
3. Select **Transfer Ownership** from the menu
4. Confirm the transfer

<Warning>
  Once ownership is transferred, you will no longer be the owner of the
  organization.
</Warning>
